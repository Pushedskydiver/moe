# BUILD_PLAN.md — moe v3

> **Sizing discipline (Alex's explicit rule):** every chunk below is small, safe, and simple — roughly one PR, one sitting, one reviewable idea. Anything that turned out bigger during drafting got split. If a chunk turns out to be complex _during implementation_, stop and split it rather than pushing through — a chunk that grows past ~300 changed LOC of real logic (excluding lockfiles/scaffolding boilerplate) is a signal to split, not a target to squeeze under.
>
> **Reading order:** `docs/VISION.md` (product truth) → `CLAUDE.md` (engineering rules) → this file (sequence). Where this file conflicts with VISION.md, VISION.md wins.
>
> **Chunk states:** `[ ]` not started · `[~]` in progress · `[x]` merged · `[GATE]` needs an Alex conversation/decision before starting — not buildable until the gate clears.

---

## How the stages ladder up

```
Stage 0  Repo skeleton + engineering spine        (nothing moe-specific yet)
Stage 1  Core domain: tickets, claims, database    (pure logic + one DB table at a time)
Stage 2  One persona process, one Slack app        (prove the loop with N=1 before N=7)
Stage 3  Intake: chat → ticket                     (the #1 failure-mode fix, on N=1)
Stage 4  GitHub + the board                        (tickets become real work on chief-clancy)
Stage 5  The full cast                             (N=1 → N=7, gated on the cast redline)
Stage 6  Team behaviour: handoffs, review, merge   (personas working together)
Stage 7  Ceremonies                                (retro, replenishment, monthly review)
Stage 8  Memory & growth                           (voice consistency over months)
```

Each stage ends with something observable working end-to-end. **The gating rule, precisely:** a stage starts when the previous stage's _exit criterion_ is met — not when every chunk in it is checked. Chunks an exit criterion doesn't cover may float across the boundary, with these hard orderings: 0.6a–0.6c/0.7 land before Stage 1's first PR; 1.5/1.6 land before Stage 5's first 5.3 sub-chunk (VISION §13.1's item 1 puts the risk-tier gate in the foundation, before any persona exists in prose); 3.6 may trail into Stage 4; 7.6 is independent and may be pulled earlier. That's the anti-scope-spiral mechanism, per VISION §13.

**On VISION §13.1's "the full cast stands up together":** this plan runs N=1 through Stages 2–4 because the plumbing those stages build (Slack transport, intake cascade, GitHub + board) is persona-independent and cheaper to debug on one process than seven — none of it is cross-persona dynamics. The cast then stands up together as one stage (Stage 5), not one-persona-at-a-time rollout. **Resolved at chunk 2.1:** this ordering is confirmed, not amended away — VISION §13.1's item 3 now says so explicitly.

## Stage 1 — Core domain: tickets, claims, database

**Exit criterion:** two concurrent fake persona processes race to claim the same ticket in an integration test; exactly one wins, every status message about it carries evidence.

- [x] **1.1 — Ticket types + Zod schemas.** `Ticket`, board statuses (`Backlog → Brief → Plan → Build → Review → Done` + `Cancelled`), severity classes, `projectKey`. Pure types + schemas + unit tests. No DB yet.
- [x] **1.2a — [GATE] Process topology × DB access ADR.** **Decided (2026-07-11): N machines, one per persona, backed by Neon Postgres.** Rejects the single-machine + SQLite-on-volume + Litestream shape this chunk originally scoped as the cheaper default — a live-verified deep-research pass found that framing wrong on three fronts: Litestream has an ongoing pattern of silent-replication-failure bugs (one fix merged then reverted, unfixed for 4+ months since, plus two more open issues in the same class — the worst failure mode for claim-critical state); Fly's own platform incident cadence makes host-level blast radius a real, not just theoretical, risk; and the real cost gap is much narrower than assumed once Neon (not Fly Managed Postgres's $38/mo floor) is the comparison point. Full reasoning, rejected alternatives (including Fly's native multi-container Machines API as a better-than-`overmind` single-machine option — still rejected), and evidence: `docs/decisions/TOPOLOGY-AND-DATABASE.md`. Written ADR with Alex. **Decision-only chunk.**
- [x] **1.2b — Database layer + tickets table.** The DB per the 1.2a ADR (Neon Postgres, N machines — one per persona), migrations tooling, the tickets table, plain CRUD. No claiming logic yet.
- [x] **1.3 — Atomic claim.** Compare-and-set claim/release on a ticket (`claimedBy` + version column). Property/concurrency tests: N racing claimants, exactly one winner, no lost updates. This is the orchestrator's load-bearing primitive (VISION §4.5) — it gets its own chunk so it's tested in isolation.
- [x] **1.4 — StatusClaim schema + composer gate.** The `{claim, toolCallId, toolOutputSnippet, timestamp}` type (VISION §7.6), plus the composer function that refuses to emit a status claim without populated evidence and falls back to "not yet verified". - [x] **1.5 — [GATE] Track-record definition ADR.** **Decided (2026-07-12):** minimum track record across directories for a multi-directory diff; preserve on git-detected rename; no transfer to a brand-new directory (Tier 2 floor); threshold **N = 5** consecutive unreverted merges. Full reasoning and evidence: `docs/decisions/CAST-ROSTER.md`. VISION §4.1 filled in + `docs/PERSONAS.md` skeleton. Written with Alex.
- [x] **2.2 — Persona process skeleton.** `apps/server` boots, loads a persona config (ID, Slack credentials), connects nothing, health-check endpoint, structured logging with secret redaction. Deployable to Fly as one machine.
- [x] **2.3 — Slack app + inbound events.** One Slack App (the first persona's), **Socket Mode** (settled by a 2026-07-04 docs verification: no public endpoint per app, event retries work the same over Socket Mode as HTTP (~6-min retry window either way), Block Kit interactivity fully supported, 10 connections per app with no workspace cap, and Slack's "HTTP for production" advice targets horizontally-scaled apps — not 7 single-instance always-on processes, which is Socket Mode's stated sweet spot). Receives DMs and channel messages, Zod-validates payloads, normalizes to an internal `InboundMessage`. Replies with a hardcoded acknowledgment — no LLM yet.
- [x] **2.4a — LLM reply loop (stateless).** Wire the **raw Anthropic Messages API** (per VISION §11's verified model-client choice — the Agent SDK's per-query subprocess spawn (~12s) and session-file accumulation make it the wrong layer for chat turns; it returns for Riley's agentic work at 6.2): a single-turn DM produces a coherent reply in a deliberately generic, non-persona placeholder voice — explicitly **not** the persona's character, which is Stage 5 behind the do-not-touch gate. The placeholder prompt lives outside `packages/agents/src/personas/*/prompt.md` so no gated path is touched. This chunk proves the client wiring end-to-end (config, turn invocation with inline tool definitions, reply extraction, error/retry/timeout, post back to Slack).
- [x] **2.4b — Thread-scoped conversation state.** Store and rehydrate per-thread history into the next turn. Keying: a DM channel is one continuous conversation; a channel/group message only accumulates history inside an explicit Slack thread, keeping unrelated people/topics in a busy channel from bleeding together — an un-threaded channel/group message stays fully stateless, same as 2.4a. Trimming: last 20 stored turns (~10 exchanges) rehydrated per call, full history retained in Postgres regardless. New `conversation_turns` table + repository in `packages/core`; `apps/server` gains its first `@moe/core`/DB dependency (an edge already legal in the dependency-direction table, now exercised). Also closes a gap Slack's own event shape creates: the message that opens a channel/group thread never carries `thread_ts` itself (only the replies that follow do), so a small in-memory root-candidate buffer backfills the opening message (and the bot's own reply to it) into a newly-formed thread's history on its first reply. **Removed at chunk 3.3**: once ambient channel/group messages stopped going through the chat-reply path at all (DMs-only), this buffer became unreachable and was deleted rather than left as dead code. A per-thread-key serialization queue prevents two concurrent messages in the same thread from racing on history fetch/persist. Migrations stay a manual pre-deploy step, not run at boot.
- [x] **2.5 — Status messages through the claim gate.** Any status claim now routes through a `report_status` tool call rather than free prose, per VISION §7.6's typed-object requirement — resolved as the correct reading of this line's own acceptance test. Composed via the 1.4 gate. Integration test: persona calls `report_status` with a claim but no backing evidence (no real work tool exists yet at Stage 2) → the message that actually reaches Slack, and gets persisted to conversation history, says "Not yet verified." Known limitation of this chunk's own enforcement, not a VISION-sanctioned exception (VISION §7.6 only names the _misgrounded_-claim gap as an accepted Tier 0/1 tradeoff, not this one): a bare free-prose claim with zero tool call at all still bypasses the gate entirely, since enforcement is a system-prompt instruction, not a mechanical block — closing that fully is Stage 5/6 territory (persona prompting, independent verifiers), not this chunk's job.
- [x] **2.6a — Cost metering substrate.** `persona_cost_daily` table + migration (`packages/core`), accumulated atomically via a real `INSERT ...
