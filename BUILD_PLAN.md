# BUILD_PLAN.md — moe v3

> **Sizing discipline (Alex's explicit rule):** every chunk below is small, safe, and simple — roughly one PR, one sitting, one reviewable idea. Anything that turned out bigger during drafting got split. If a chunk turns out to be complex _during implementation_, stop and split it rather than pushing through — a chunk that grows past ~300 changed LOC of real logic (excluding lockfiles/scaffolding boilerplate) is a signal to split, not a target to squeeze under.
>
> **Reading order:** `docs/VISION.md` (product truth) → `CLAUDE.md` (engineering rules) → this file (sequence). Where this file conflicts with VISION.md, VISION.md wins.
>
> **Chunk states:** `[ ]` not started · `[~]` in progress · `[x]` merged · `[GATE]` needs an Alex conversation/decision before starting — not buildable until the gate clears.

---

## How the stages ladder up

```
Stage 0  Repo skeleton + engineering spine        (nothing moe-specific yet)
Stage 1  Core domain: tickets, claims, database    (pure logic + one DB table at a time)
Stage 2  One persona process, one Slack app        (prove the loop with N=1 before N=7)
Stage 3  Intake: chat → ticket                     (the #1 failure-mode fix, on N=1)
Stage 4  GitHub + the board                        (tickets become real work on chief-clancy)
Stage 5  The full cast                             (N=1 → N=7, gated on the cast redline)
Stage 6  Team behaviour: handoffs, review, merge   (personas working together)
Stage 7  Ceremonies                                (retro, replenishment, monthly review)
Stage 8  Memory & growth                           (voice consistency over months)
```

Each stage ends with something observable working end-to-end. **The gating rule, precisely:** a stage starts when the previous stage's _exit criterion_ is met — not when every chunk in it is checked. Chunks an exit criterion doesn't cover may float across the boundary, with these hard orderings: 0.6a–0.6c/0.7 land before Stage 1's first PR; 1.5/1.6 land before Stage 5's first 5.3 sub-chunk (VISION §13.1.1 puts the risk-tier gate in the foundation, before any persona exists in prose); 3.6 may trail into Stage 4; 7.6 is independent and may be pulled earlier. That's the anti-scope-spiral mechanism, per VISION §13.

**On VISION §13.1's "the full cast stands up together":** this plan runs N=1 through Stages 2–4 because the plumbing those stages build (Slack transport, intake cascade, GitHub + board) is persona-independent and cheaper to debug on one process than seven — none of it is cross-persona dynamics. The cast then stands up together as one stage (Stage 5), not one-persona-at-a-time rollout. §13.1.3 can be read as wanting the cast immediately after the §13.1.1 foundation, before Stage 4. **Open question (settle at 2.1):** Alex confirms this ordering and VISION §13.1 gets a matching amendment, or Stage 5 moves ahead of Stage 4.

## Stage 1 — Core domain: tickets, claims, database

**Exit criterion:** two concurrent fake persona processes race to claim the same ticket in an integration test; exactly one wins, every status message about it carries evidence.

- [ ] **1.1 — Ticket types + Zod schemas.** `Ticket`, board statuses (`Backlog → Brief → Plan → Build → Review → Done` + `Cancelled`), severity classes, `projectKey`. Pure types + schemas + unit tests. No DB yet.
- [ ] **1.2a — [GATE] Process topology × DB access ADR.** One machine running all N persona processes vs one machine per persona — and the DB shape that supports the choice. The constraint set was verified against Fly's primary docs (2026-07-04) and is fully known today: a Fly volume attaches to exactly one machine (no multi-attach product exists); Fly `[processes]` groups spawn _separate machines_ per group, so the one-machine shape uses a supervisor (overmind, per Fly's own multi-process guide) inside the image; **LiteFS is disqualified** — single-writer primary/replica (N machines can't all do compare-and-set claims) and support-orphaned — so the N-machine branch forces Postgres (Fly Managed Postgres $38/mo floor, or Neon usage-based). The one-machine + SQLite-on-volume + Litestream-backup shape is the only one where SQLite CAS keeps working, and the cheapest (~$12–25/mo vs ~$20–80/mo); its cost is blast radius — one crash/host failure downs every persona together. That trade is the actual decision. 1.2b's DB layer plus 1.3's concurrency tests are built against the answer — the decision closes here, once, rather than betting at Stage 1 and re-deciding at Stage 5. Written ADR with Alex. **Decision-only chunk.**
- [ ] **1.2b — Database layer + tickets table.** The DB per the 1.2a ADR (single-machine default: SQLite on the Fly volume, same as v2), migrations tooling, the tickets table, plain CRUD. No claiming logic yet.
- [ ] **1.3 — Atomic claim.** Compare-and-set claim/release on a ticket (`claimedBy` + version column). Property/concurrency tests: N racing claimants, exactly one winner, no lost updates. This is the orchestrator's load-bearing primitive (VISION §4.5) — it gets its own chunk so it's tested in isolation.
- [ ] **1.4 — StatusClaim schema + composer gate.** The `{claim, toolCallId, toolOutputSnippet, timestamp}` type (VISION §7.6), plus the composer function that refuses to emit a status claim without populated evidence and falls back to "not yet verified". Pure logic, heavily unit-tested. **This is failure-mode-#2's fix and it ships before any persona exists.**
- [ ] **1.5 — [GATE] Track-record definition ADR.** Resolve VISION §8.1's open definitional question (multi-directory diffs, renames, new directories, the threshold N) as a written ADR with Alex. **Decision-only chunk — no code.** Blocks 1.6. May float later than Stage 1, but must land before Stage 5's first 5.3 sub-chunk — VISION §13.1.1 puts the risk-tier gate before any persona exists in prose.
- [ ] **1.6 — Risk-tier classifier.** `classifyRiskTier(diffMeta) → Tier 0–3` as a pure function per the ADR from 1.5 + VISION §8.1's table. Property-based tests. No enforcement wiring yet — just the classifier.

- [ ] **2.6a — Cost metering substrate.** Token/cost tracking per persona per day: the persisted table + migration, and per-turn accounting hooks threaded through the LLM call path. Accumulate and expose the numbers, nothing more. Small now, painful to retrofit later.
