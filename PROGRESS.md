**Chunk 2.4b — thread-scoped conversation state** (PR #32): new `conversation_turns` table + repository in `packages/core` (`apps/server`'s first `@moe/core`/DB dependency — an edge already legal in the dependency-direction table, now exercised for the first time). Keying resolved with Alex ahead of implementation: a DM channel is one continuous conversation regardless of Slack's own `thread_ts`; a channel/group message only accumulates history inside an explicit Slack thread, so unrelated people/topics in a busy channel don't bleed together; trimming at the last 20 stored turns (~10 exchanges) rehydrated per call, full history retained in Postgres regardless. Also closes a gap the implementation plan's own `spec-grill` pass surfaced: Slack's `message` event never carries `thread_ts` on the message that opens a thread, only on the replies that follow, so a small in-memory root-candidate buffer backfills the opening message (and the bot's own reply to it) into a newly-formed thread's history on its first reply. A per-thread-key serialization queue prevents two concurrent messages in the same thread racing on history fetch/persist. Migrations stay a manual pre-deploy step, not run at `apps/server` boot.

**Process arc, the fullest yet on a single chunk**: the implementation plan went through 3 rounds of `spec-grill`'s two-phase discipline before any code moved — R1 discovery found 3 BLOCKING + 4 MATERIAL + 3 LOW issues (a `resetDatabase()` gap that would've broken migration tests, an `eslint max-params` violation, an unhandled concurrency race, an under-scoped file move, among others, all folded); R2 verification confirmed 8 of 10 folds and found 2 NEW MATERIAL issues inside the fold's own new content (an incorrect claim about Anthropic's API behavior, an unaddressed race in the new backfill buffer); a scoped R3 confirmed those fixes but found the accompanying tests only half-covered the claim, also fixed. Post-implementation, DA review found 2 real MATERIAL bugs neither `spec-grill` nor the author's own testing had caught: the thread-queue was keyed on `threadKey` alone — the constant `'dm'` for every DM — collapsing every DM conversation in the whole persona process into one serialization lane instead of one per conversation; and `exitAndCloseServer` never closed the Postgres pool, reintroducing the exact "exit doesn't actually take effect" bug class chunk 2.3 already fixed once, just for a different resource. Both fixed and regression-tested via the Prove-It Pattern (confirmed failing/hanging against the prior code before the fix, not just passing after). A fresh-context R2 independently mutation-tested both fixes — reverting each and confirming the regression tests actually catch it — rather than trusting the commit message; it also caught one LOW/FYI (a theoretical colon-collision in the new queue key), closed with `JSON.stringify` encoding.
