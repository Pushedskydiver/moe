**Claim-then-act fallback fix** ([PR #51](https://github.com/Pushedskydiver/moe/pull/51), not a numbered `BUILD_PLAN.md` chunk — Alex asked for it directly, same untracked-follow-up shape as [PR #47](https://github.com/Pushedskydiver/moe/pull/47)'s Socket Mode idempotency guard): closes a structural gap DA review first flagged at chunk 3.4b-ii and found recurring a second time at chunk 3.5 — `commitAsTicket`, `draftFromConfirmingQuestion`, and `logStaleQuestionsAsSilent` each atomically claim a row, then perform a second action that can fail, permanently burning the claim with nothing to show for it. The architectural pass (before any code moved) found a **fourth** call site DA's own tracked findings never named — `logConfirmingQuestionAsNo` — by grepping every claim call site in `apps/server/src` rather than trusting the three-site framing; scope-confirmed in via `AskUserQuestion`. The pass also found the fix _shouldn't_ be uniform across all four sites, the framing the original ask implied as "likely": three sites (claim → a second pure DB write, nothing external in between) can be made **fully atomic** via a single Kysely transaction — this codebase's first runtime use of an explicit transaction — closing the gap completely, since a downstream-write failure rolls the claim back too. The fourth site (`draftFromConfirmingQuestion`, claim → Anthropic + Slack calls → a DB write) can't be made transactional at all — a transaction can't hold open across two slow external calls, and a rollback can't un-send an already-posted Slack message — so it keeps its own explicit claim and gets a best-effort `review_queue` fallback write instead, with an explicitly documented residual gap.

This design was verified via a full two-phase `spec-grill` before any code moved — the first time this project has used `spec-grill` on a refactor spec rather than a rule-promotion doc. R1 discovery found 2 BLOCKING (the original design would have re-introduced a stale-title race an earlier chunk's own TSDoc had documented fixing, by accepting a pre-claim caller-supplied title instead of deriving it from the post-claim row inside the transaction; the DI-wiring plan targeted the wrong shared type, which would have leaked a reaction-outcome-only concern into the unrelated message-handling path) and 3 MATERIAL (flattening two error union types into one would have lost the existing info-vs-error log-level distinction; a new `outcomeReason` value would have silently vanished from the sweep's own digest body despite type-checking cleanly into a different, compiler-checked list in the same file; two precedent citations about `migrate.ts` and Kysely's own transaction API were factually wrong) — all folded, and R_n verification (a distinct, skeptical-not-discovery round) independently re-confirmed each fix against the real repo and found the design at a genuine nit-floor.
