# Glossary

Ubiquitous language for moe — use consistently in code, docs, commits, and persona prompts. Shape adapted from chief-clancy's own `GLOSSARY.md` (category-grouped term tables, each entry linking to its authoritative doc section); the terms themselves are moe's own, not a port — chief-clancy's terminology is almost entirely specific to its own installer/board/pipeline system, which moe doesn't have.

`docs/VISION.md` Appendix B lists several terms as "carried forward unchanged from the previous design's glossary." That prior design "is not retained in this repo as a checkable reference file" (`docs/VISION.md`'s own front matter) — the entries below define those terms from where moe's own current docs actually use them, or from their standard industry meaning where moe hasn't specified anything narrower yet. Where a term is still an open question rather than a settled definition, the entry says so.

## Anti-fabrication (VISION §7.6)

| Term                        | Definition                                                                                                                                                                           |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Execution hallucination** | An agent claiming to have completed a sub-stage of work it did not actually complete. The academic name for the second of moe v3's three named failure modes (`docs/VISION.md` §0).  |
| **Evidence-gated claim**    | A status statement a persona is structurally prevented from emitting unless it's backed by a real `toolCallId`/`toolOutputSnippet` pair.                                             |
| **Ungrounded fabrication**  | A claim with no tool-call evidence at all — closed by the evidence-gated-claim schema itself.                                                                                        |
| **Misgrounded fabrication** | A claim backed by real but irrelevant evidence. Closed only for Tier 2/3 claims (see Autonomy & trust, below), by independent verification — the schema gate alone doesn't catch it. |

## Autonomy & trust (VISION §8)

| Term             | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Risk tier**    | The path × change-shape classifier that gates how much review a change needs before merge (`docs/VISION.md` §8.1) — replaces the old time-gated Probation model entirely.                                                                                                                                                                                                                                                                                  |
| **Tier 0**       | Auto-merge immediately. All CI green; no sensitive path touched; diff below a size threshold; docs/config/lint-only or a non-major devDependency bump.                                                                                                                                                                                                                                                                                                     |
| **Tier 1**       | Fast single-approve, same-day. Logic change with an accompanying test, in a path the agent has a track record on.                                                                                                                                                                                                                                                                                                                                          |
| **Tier 2**       | Standard review. Any logic change without an accompanying test, a path with no track record, or a diff crossing package boundaries.                                                                                                                                                                                                                                                                                                                        |
| **Tier 3**       | Mandatory named-owner review, cannot be satisfied by the requesting persona or the human who dispatched it. Auth, payments, PII/secrets, CI/CD config, migrations, or any destructive operation, regardless of track record — a hard floor autonomy doesn't earn past.                                                                                                                                                                                     |
| **Track record** | A path's history of unreverted merges; shifts a change down at most one tier, and never into Tier 3. A diff spanning multiple directories takes the _minimum_ track record across them; a directory rename/move preserves it when git detects the rename; a brand-new directory never inherits it (Tier 2 floor); the threshold is **N = 5** consecutive unreverted merges (`docs/decisions/TRACK-RECORD-DEFINITION.md`, resolving `docs/VISION.md` §8.1). |
| **Trust Level**  | The previous design's time-gated Probation → Trusted → Veteran ladder. **Superseded** by the risk-tier model above — there is no more Probation phase gating all pulls.                                                                                                                                                                                                                                                                                    |
| **Veto signal**  | Alex's 🛑 reaction, which holds a merge. Carries over conceptually from the previous design (`docs/VISION.md` §8.2); not yet built.                                                                                                                                                                                                                                                                                                                        |

| **Review-queue sweep**               | BUILD_PLAN 3.5's own reader for **Review queue** (above) — VISION §5.2's "nothing is silently eaten" backstop, finally given a human-facing surface. | **Draft-outcome tracking**           | BUILD_PLAN 3.6's own counting layer over **Reaction-gate outcomes** (above) — VISION §5.4's named production metric for the whole intake cascade ("the rate of ignored/rejected drafts, not an academic precision score"). BUILD_PLAN's own three-bucket text (✅-committed / 🔁-redone / ignored) doesn't map cleanly onto the real four-way outcome space (✅/📦/🔁/no reaction), and 🔁 isn't terminal — a redone draft can still later be ✅/📦'd. Three bucket-definition questions resolved with Alex via `AskUserQuestion`: `'committed'` folds in both ✅ and 📦 (both mean "accepted as real work"); `'redone'` means still-open **and** regenerated at least once (no age qualifier — a redo is itself an explicit human action); `'ignored'` requires never-redone **and** older than a 24-hour threshold (reusing **Review-queue sweep**'s own silence-threshold reasoning, so a just-posted draft isn't miscounted before anyone's had a fair chance to react — a still-open, never-redone, too-young draft falls into none of the three buckets). Separately, counts are lifetime-cumulative, not windowed like `review_queue`'s own entries — VISION §5.4 names the ignored/rejected-draft rate as the metric to watch, which favors the full accumulated trend over a noisy per-window slice. Built as a `redo_count` column on `pending_ticket_drafts` (migration `0012_add_pending_ticket_drafts_redo_count.sql`, `Generated<number>`, excluded from `pendingTicketDraftSchema`'s own pure domain shape — same reasoning `ticketSchema` excludes `version`/`claimedBy`), incremented inside `updatePendingTicketDraftContent`'s existing UPDATE, and `getDraftOutcomeCounts` (`packages/core`'s `intake/draft-outcome-counts.ts`), a single aggregate query using Postgres `FILTER (WHERE ...)` clauses rather than three round-trips. Surfaced as a `📊 Draft outcomes (all time): N committed, M redone (open), K ignored — R% acceptance rate` line in the **Review-queue sweep**'s own digest (rate computed over `committed`/`ignored` only, excluding still-open `redone` drafts) — rides along with an existing digest post rather than independently triggering one, since a lifetime-cumulative count is almost never zero and would otherwise defeat the sweep's "quiet when nothing new" behavior. Scoped to `origin: 'high-band'` only (DA review, chunk 3.6) — `pending_ticket_drafts` is a shared table for both High-band auto-drafts and Mid-band 👍-confirmed drafts (via `postAndPersistDraft`, since chunk 3.4b-ii), and counting both together skewed the reported rate upward; a new `origin` column (migration `0013_add_pending_ticket_drafts_origin.sql`) distinguishes them, confirmed against VISION §5.2's own text before implementing rather than trusting the finding's own framing.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
