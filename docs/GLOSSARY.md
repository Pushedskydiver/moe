# Glossary

Ubiquitous language for moe — use consistently in code, docs, commits, and persona prompts. Shape adapted from chief-clancy's own `GLOSSARY.md` (category-grouped term tables, each entry linking to its authoritative doc section); the terms themselves are moe's own, not a port — chief-clancy's terminology is almost entirely specific to its own installer/board/pipeline system, which moe doesn't have.

`docs/VISION.md` Appendix B lists several terms as "carried forward unchanged from the previous design's glossary." That prior design "is not retained in this repo as a checkable reference file" (`docs/VISION.md`'s own front matter) — the entries below define those terms from where moe's own current docs actually use them, or from their standard industry meaning where moe hasn't specified anything narrower yet. Where a term is still an open question rather than a settled definition, the entry says so.

## Anti-fabrication (VISION §7.6)

| Term                        | Definition                                                                                                                                                                           |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Execution hallucination** | An agent claiming to have completed a sub-stage of work it did not actually complete. The academic name for the second of moe v3's three named failure modes (`docs/VISION.md` §0).  |
| **Evidence-gated claim**    | A status statement a persona is structurally prevented from emitting unless it's backed by a real `toolCallId`/`toolOutputSnippet` pair.                                             |
| **Ungrounded fabrication**  | A claim with no tool-call evidence at all — closed by the evidence-gated-claim schema itself.                                                                                        |
| **Misgrounded fabrication** | A claim backed by real but irrelevant evidence. Closed only for Tier 2/3 claims (see Autonomy & trust, below), by independent verification — the schema gate alone doesn't catch it. |

## Autonomy & trust (VISION §8)

| Term             | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Risk tier**    | The path × change-shape classifier that gates how much review a change needs before merge (`docs/VISION.md` §8.1) — replaces the old time-gated Probation model entirely.                                                                                                                                                                                                                                                                                  |
| **Tier 0**       | Auto-merge immediately. All CI green; no sensitive path touched; diff below a size threshold; docs/config/lint-only or a non-major devDependency bump.                                                                                                                                                                                                                                                                                                     |
| **Tier 1**       | Fast single-approve, same-day. Logic change with an accompanying test, in a path the agent has a track record on.                                                                                                                                                                                                                                                                                                                                          |
| **Tier 2**       | Standard review. Any logic change without an accompanying test, a path with no track record, or a diff crossing package boundaries.                                                                                                                                                                                                                                                                                                                        |
| **Tier 3**       | Mandatory named-owner review, cannot be satisfied by the requesting persona or the human who dispatched it. Auth, payments, PII/secrets, CI/CD config, migrations, or any destructive operation, regardless of track record — a hard floor autonomy doesn't earn past.                                                                                                                                                                                     |
| **Track record** | A path's history of unreverted merges; shifts a change down at most one tier, and never into Tier 3. A diff spanning multiple directories takes the _minimum_ track record across them; a directory rename/move preserves it when git detects the rename; a brand-new directory never inherits it (Tier 2 floor); the threshold is **N = 5** consecutive unreverted merges (`docs/decisions/TRACK-RECORD-DEFINITION.md`, resolving `docs/VISION.md` §8.1). |
| **Trust Level**  | The previous design's time-gated Probation → Trusted → Veteran ladder. **Superseded** by the risk-tier model above — there is no more Probation phase gating all pulls.                                                                                                                                                                                                                                                                                    |
| **Veto signal**  | Alex's 🛑 reaction, which holds a merge. Carries over conceptually from the previous design (`docs/VISION.md` §8.2); not yet built.                                                                                                                                                                                                                                                                                                                        |

## Team & ceremonies (VISION §3, §6)

| Term                   | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ScrumBan**           | Moe's process model: Kanban flow (continuous pull, WIP limits) plus Scrum-style ceremonies layered on top (`docs/VISION.md` §3.1).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Ceremony**           | A structured, scheduled team ritual (e.g. weekly replenishment, monthly review) — `docs/VISION.md` §3.2; specific ceremony formats live in `docs/CEREMONIES.md` once written (do-not-touch surface, `CLAUDE.md`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Brief**              | A work-scoping artifact a persona produces before implementation (illustrated in `docs/VISION.md` §1.3's day-in-the-life dialogue: "Sarah's brief for the packages issue"). Not yet formally specified as a schema or command — narrative usage only as of Stage 0.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **EOD digest**         | The end-of-day summary a persona posts to the team channel (`docs/VISION.md` §6.5), standing in for a standup — Alex is continuously present in the channel, so a bolted-on standup on top would be redundant (§3.2).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Core hours**         | The window during which proactive persona behavior (sends, intake drafts) is allowed; off-hours defers to the next window. Settled at `BUILD_PLAN.md` chunk 2.7a (Alex confirmed, not inferred): 08:30–17:00 Europe/London Mon–Fri, weekends and GOV.UK bank holidays (england-and-wales division, cached with a serve-stale-on-failure fallback) excluded. Direct DM replies are exempt — they're reactive engagement, not Moe acting unprompted, so §14's rest rule doesn't reach them. `docs/VISION.md` §6.4 delegates the parameters here.                                                                                                                                                                                                                                                                                                                            |
| **Away-detection**     | Reading Alex's Slack status text/emoji against a keyword list to suppress off-hours proactive behavior further. Settled at `BUILD_PLAN.md` chunk 2.7b (Alex confirmed, not inferred): text keywords `away`/`holiday`/`OOO` (whole-word, case-insensitive — a bare substring match would false-positive inside `giveaway`/`runaway`); emoji shortcodes `palm_tree`/`beach_with_umbrella`/`airplane`/`face_with_thermometer`/`thermometer`/`hospital` (verified against the real `iamcal/emoji-data` short-name database, not guessed — `beach_with_umbrella` specifically, not the more obvious-sounding `beach_umbrella`). Deliberately excludes momentary-busy signals (🔴 do-not-disturb, 🏠 WFH) that don't reliably mean extended absence. First real consumer is 7.2b; this chunk builds `isAway`/`fetchSlackStatus` (`packages/slack`) as reusable primitives only. |
| **Classes of Service** | A Kanban capacity concept (work-item categories with different WIP/priority treatment). Single-project classes of service are in-scope but **not yet resolved** (`docs/VISION.md` §3.1) — must be settled before board code is written.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **WSJF**               | Weighted Shortest Job First — a prioritization technique (cost of delay ÷ job size). Named in VISION Appendix B as carried-forward vocabulary; not yet wired into any moe mechanism.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

## Intake & triage (VISION §5)

| Term                          | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Stage 0 — channel scoping** | The coarse, per-surface filter that runs before the classifier ever sees a message (`docs/VISION.md` §5.2) — "a message only enters the pipeline if it's in a channel/DM the team already treats as work-relevant." Built at `BUILD_PLAN.md` chunk 3.2 as `isSurfaceInScope`/`ChannelScopeConfig` (`packages/core`): a DM to a named persona is always in scope (§5.3 — already unambiguous, handled by that persona's own app without Sarah in the loop; this predicate short-circuits Stage 0 for that case rather than gating on it, since §5.3 itself doesn't speak to Stage 0 specifically); a channel is in scope only if its ID is in the configured work-relevant set (§6.1's roster: `#moe-team`/`#moe-incidents`/`#moe-research` in, `#moe-random` out). Coarse by design — it removes whole channels of banter without keyword rules, leaving message-level precision to Stage 1. Wired into `apps/server`'s inbound-message handler at chunk 3.3 (`parseChannelScopeConfig`, `packages/agents`, resolves real Slack channel IDs from `MOE_WORK_RELEVANT_CHANNEL_IDS`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Stage 1 classifier**        | The cheap, per-message model call that scores whether an in-scope message describes work (`docs/VISION.md` §5.2) — numeric, not binary, so it can drive three-band routing rather than a yes/no gate. Scoring method and thresholds settled at `BUILD_PLAN.md` chunk 3.1 (`docs/decisions/STAGE-1-CLASSIFIER.md`): one bundled structured-output call returning a single 0–100 confidence integer, on Claude Haiku 4.5 — no log-probs (the Claude API doesn't expose them, on any model) involved — prefill isn't relevant either, since it's confirmed dead only on other current models, not on Haiku 4.5 itself. Built as `classifyMessageConfidence` at chunk 3.3 (`packages/agents`, `zodOutputFormat` + `.messages.parse()`); wired into every in-scope ambient channel/group message via `apps/server`. As of chunk 3.4a-i, a High-band score additionally drives ticket-draft composition (see **High-band auto-draft**, below) — Mid/Low bands still just log the score, until 3.4b/3.4c wire their own actions. Gated by the same per-persona monthly cost cap as the DM chat-reply path (`checkCostCapAndAlert`, `haikuCostUsdMicros`, BUILD_PLAN 2.6a/2.6b) — a real, billed Anthropic call regardless of which model or call site it's on.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **Confidence band**           | The High/Mid/Low routing outcome derived from the Stage 1 classifier's numeric score (`docs/VISION.md` §5.2) — High ≥ 70 auto-drafts a reversible ticket, Mid (35–69) asks a confirming question, Low (< 35) logs silently to the review queue. Thresholds calibrated at `BUILD_PLAN.md` chunk 3.1 against a live eval, not chosen from theory. Built as `classifyConfidenceBand` at chunk 3.4a-i (`packages/core`, a pure `number → ConfidenceBand` function) — only the High band is wired to a real action so far (**High-band auto-draft**, below); Mid/Low still just log the classification, same as before this chunk.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **High-band auto-draft**      | The action a High-band confidence score triggers (`docs/VISION.md` §5.2) — an LLM composes a link-only ticket draft (title + body, no invented detail beyond the source message) from the classified message. Built at `BUILD_PLAN.md` chunk 3.4a-i as `composeTicketDraft` (`packages/agents`, Claude Sonnet 5, `zodOutputFormat` + `.messages.parse()`), wired into `handleAmbientChannelMessage` (`apps/server`). Shadow/log-only for now (Alex confirmed via `AskUserQuestion`, not inferred, same "shadow only" precedent as chunk 3.3 for the same class of risk — chunk 6.5a-i's own plan later follows this same pattern too, per its own text): the draft and its intended 📦/🔁/✅ reaction legend (park to Backlog / redo / commit as a ticket) are only logged, not actually posted to Slack, until 3.4a-iii's situational-appropriateness gate (VISION §9) exists in front of it. Respects the 2.7a operating-rhythm guard (`evaluateOperatingRhythm`) — 2.7a's own settled off-hours policy names "intake drafts" specifically as deferring to the next core-hours window, unlike the classification step itself, which that guard doesn't gate. Gated by its own fresh per-persona cost-cap check and `sonnetCostUsdMicros` accounting, same mechanism as every other real, billed call site in this cascade.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Reaction-gate outcomes**    | The three deterministic actions the 📦/🔁/✅ reaction legend maps to (`docs/VISION.md` §5.4's trust-erosion rule: routing decisions with real consequences stay off the LLM layer, not just the drafting step). Built at `BUILD_PLAN.md` chunk 3.4a-ii: `classifyReactionOutcome` (`packages/slack`, a pure Slack-reaction-short-name → outcome mapping — short-names verified against the real `iamcal/emoji-data` database, not guessed: ✅ is `white_check_mark`, 🔁 is `repeat`, 📦 is `package`), a `pending_ticket_drafts` table + repository (`packages/core`, the "parent-message state" tracking which draft a Slack message's reactions belong to — `resolvePendingTicketDraft`'s atomic claim, same CAS shape as chunk 1.3's `claimTicket`, guards against a reaction double-firing the same outcome twice), and three outcome-path functions (`apps/server`'s `reaction-outcome-actions.ts`): ✅ commits the draft as a real `Brief` ticket, 📦 parks it to an untriaged `Backlog` ticket (both `projectKey: 'chief-clancy'`, VISION §3.4's single-project scope; `severity: 'Medium'`, a fixed placeholder Alex confirmed via `AskUserQuestion` until a real triage signal exists), and 🔁 recomposes a fresh draft from the _original_ source message via `composeTicketDraft` (its own fresh cost-cap check and `sonnetCostUsdMicros` accounting) and updates the pending draft in place, leaving it open for a further reaction. Built as real, tested primitives with no live consumer yet (Alex confirmed via `AskUserQuestion`) — no Socket Mode `reaction_added` listener is registered against a real process, and 3.4a-i's own draft composition doesn't persist a `pending_ticket_drafts` row either, so nothing in production can yet produce a reaction this gate would see; both halves of the real end-to-end loop wait for 3.4a-iii's situational-appropriateness gate, per that chunk's own BUILD_PLAN text. |

## Review & process (`docs/DEVELOPMENT.md`, `docs/DA-REVIEW.md`)

| Term                         | Definition                                                                                                                                                                                                                                                                                          |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Chunk**                    | A single unit of `BUILD_PLAN.md` work — roughly one PR, one sitting, one reviewable idea, per `BUILD_PLAN.md`'s own sizing-discipline note.                                                                                                                                                         |
| **Stage**                    | A group of chunks in `BUILD_PLAN.md` ending in something observable working end-to-end.                                                                                                                                                                                                             |
| **[GATE] chunk**             | A chunk that needs an Alex conversation before it can proceed — flagged explicitly in `BUILD_PLAN.md`, never guessed past.                                                                                                                                                                          |
| **DA review**                | Devil's-advocate review, dispatched from a fresh context, checked against `docs/DA-REVIEW.md` + `docs/CONVENTIONS.md` + `docs/RATIONALIZATIONS.md` + `docs/REVIEW-PATTERNS.md`.                                                                                                                     |
| **spec-grill**               | Two-phase adversarial review of a spec/plan/rationale doc before code moves, per `docs/DEVELOPMENT.md` §Two-phase grill discipline.                                                                                                                                                                 |
| **copilot-surrogate**        | Factual-claim reviewer, mandatory on `fix(docs)` commits and on any PR touching a blast-radius doc, exceeding 50 LOC, or making a factual claim about the rest of the repo or an external library (`docs/decisions/REVIEW-GATE-DISCRETION.md`), reading the full PR diff at HEAD (not diff-scoped). |
| **Do-not-touch list**        | Surfaces requiring Alex's explicit approval before editing (persona prompts, `docs/CEREMONIES.md`, `docs/VISION.md` §2/§14/§4.1) — `CLAUDE.md` §Non-obvious constraints.                                                                                                                            |
| **NOTICED BUT NOT TOUCHING** | The self-review discipline of listing an out-of-scope improvement rather than fixing it inline (`docs/SELF-REVIEW.md`).                                                                                                                                                                             |
| **Blast-radius doc**         | A policy-tier doc requiring Alex's merge (not self-merge), listed in `docs/GIT.md`. This doc, `ARCHITECTURE.md`, and `decisions/` are deliberately NOT on that list, matching chief-clancy's own `docs/INDEX.md` precedent (its own reference-tier docs are auto-merge-eligible).                   |
| **ADR**                      | Architecture Decision Record — the general industry term for a short, dated record of an architectural choice and its rationale. Moe's equivalent shape lives in `docs/decisions/`.                                                                                                                 |

## Tools & general terms

| Term                    | Definition                                                                                                                                                                                          |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **HITL**                | Human-in-the-loop — a step requiring human judgment before proceeding.                                                                                                                              |
| **MCP**                 | Model Context Protocol — Anthropic's protocol for connecting tools to Claude. Per-persona tool allowlists and the CLI-vs-MCP decision rule are specified in `BUILD_PLAN.md` (`docs/VISION.md` §11). |
| **Four-eyes principle** | The general security/engineering practice of requiring two people (or an agent plus a human, or two independent agents) to approve a sensitive action before it takes effect.                       |
