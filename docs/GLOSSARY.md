# Glossary

Ubiquitous language for moe — use consistently in code, docs, commits, and persona prompts. Shape adapted from chief-clancy's own `GLOSSARY.md` (category-grouped term tables, each entry linking to its authoritative doc section); the terms themselves are moe's own, not a port — chief-clancy's terminology is almost entirely specific to its own installer/board/pipeline system, which moe doesn't have.

`docs/VISION.md` Appendix B lists several terms as "carried forward unchanged from the previous design's glossary." That prior design "is not retained in this repo as a checkable reference file" (`docs/VISION.md`'s own front matter) — the entries below define those terms from where moe's own current docs actually use them, or from their standard industry meaning where moe hasn't specified anything narrower yet. Where a term is still an open question rather than a settled definition, the entry says so.

## Anti-fabrication (VISION §7.6)

| Term                        | Definition                                                                                                                                                                           |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Execution hallucination** | An agent claiming to have completed a sub-stage of work it did not actually complete. The academic name for the second of moe v3's three named failure modes (`docs/VISION.md` §0).  |
| **Evidence-gated claim**    | A status statement a persona is structurally prevented from emitting unless it's backed by a real `toolCallId`/`toolOutputSnippet` pair.                                             |
| **Ungrounded fabrication**  | A claim with no tool-call evidence at all — closed by the evidence-gated-claim schema itself.                                                                                        |
| **Misgrounded fabrication** | A claim backed by real but irrelevant evidence. Closed only for Tier 2/3 claims (see Autonomy & trust, below), by independent verification — the schema gate alone doesn't catch it. |

## Autonomy & trust (VISION §8)

| Term             | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Risk tier**    | The path × change-shape classifier that gates how much review a change needs before merge (`docs/VISION.md` §8.1) — replaces the old time-gated Probation model entirely.                                                                                                                                                                                                                                                                                  |
| **Tier 0**       | Auto-merge immediately. All CI green; no sensitive path touched; diff below a size threshold; docs/config/lint-only or a non-major devDependency bump.                                                                                                                                                                                                                                                                                                     |
| **Tier 1**       | Fast single-approve, same-day. Logic change with an accompanying test, in a path the agent has a track record on.                                                                                                                                                                                                                                                                                                                                          |
| **Tier 2**       | Standard review. Any logic change without an accompanying test, a path with no track record, or a diff crossing package boundaries.                                                                                                                                                                                                                                                                                                                        |
| **Tier 3**       | Mandatory named-owner review, cannot be satisfied by the requesting persona or the human who dispatched it. Auth, payments, PII/secrets, CI/CD config, migrations, or any destructive operation, regardless of track record — a hard floor autonomy doesn't earn past.                                                                                                                                                                                     |
| **Track record** | A path's history of unreverted merges; shifts a change down at most one tier, and never into Tier 3. A diff spanning multiple directories takes the _minimum_ track record across them; a directory rename/move preserves it when git detects the rename; a brand-new directory never inherits it (Tier 2 floor); the threshold is **N = 5** consecutive unreverted merges (`docs/decisions/TRACK-RECORD-DEFINITION.md`, resolving `docs/VISION.md` §8.1). |
| **Trust Level**  | The previous design's time-gated Probation → Trusted → Veteran ladder. **Superseded** by the risk-tier model above — there is no more Probation phase gating all pulls.                                                                                                                                                                                                                                                                                    |
| **Veto signal**  | Alex's 🛑 reaction, which holds a merge. Carries over conceptually from the previous design (`docs/VISION.md` §8.2); not yet built.                                                                                                                                                                                                                                                                                                                        |

| **Core hours**         | The window during which proactive persona behavior (sends, intake drafts) is allowed; off-hours defers to the next window. | **Away-detection**     | Reading Alex's Slack status text/emoji against a keyword list to suppress off-hours proactive behavior further. Settled at `BUILD_PLAN.md` chunk 2.7b (Alex confirmed, not inferred): text keywords `away`/`holiday`/`OOO` (whole-word, case-insensitive — a bare substring match would false-positive inside `giveaway`/`runaway`); emoji shortcodes `palm_tree`/`beach_with_umbrella`/`airplane`/`face_with_thermometer`/`thermometer`/`hospital` (verified against the real `iamcal/emoji-data` short-name database, not guessed — `beach_with_umbrella` specifically, not the more obvious-sounding `beach_umbrella`). Deliberately excludes momentary-busy signals (🔴 do-not-disturb, 🏠 WFH) that don't reliably mean extended absence. First real consumer is 7.2b; this chunk builds `isAway`/`fetchSlackStatus` (`packages/slack`) as reusable primitives only. |
| **Classes of Service** | A Kanban capacity concept (work-item categories with different WIP/priority treatment). Single-project classes of service are in-scope but **not yet resolved** (`docs/VISION.md` §3.1) — must be settled before board code is written.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **WSJF**               | Weighted Shortest Job First — a prioritization technique (cost of delay ÷ job size). Named in VISION Appendix B as carried-forward vocabulary; not yet wired into any moe mechanism.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

| **Stage 0 — channel scoping**        | The coarse, per-surface filter that runs before the classifier ever sees a message (`docs/VISION.md` §5.2) — "a message only enters the pipeline if it's in a channel/DM the team already treats as work-relevant." Wired into `apps/server`'s inbound-message handler at chunk 3.3 (`parseChannelScopeConfig`, `packages/agents`, resolves real Slack channel IDs from `MOE_WORK_RELEVANT_CHANNEL_IDS`). | **Stage 1 classifier**               | The cheap, per-message model call that scores whether an in-scope message describes work (`docs/VISION.md` §5.2) — numeric, not binary, so it can drive three-band routing rather than a yes/no gate. Scoring method and thresholds settled at `BUILD_PLAN.md` chunk 3.1 (`docs/decisions/STAGE-1-CLASSIFIER.md`): one bundled structured-output call returning a single 0–100 confidence integer, on Claude Haiku 4.5 — no log-probs (the Claude API doesn't expose them, on any model) involved — prefill isn't relevant either, since it's confirmed dead only on other current models, not on Haiku 4.5 itself. Built as `classifyMessageConfidence` at chunk 3.3 (`packages/agents`, `zodOutputFormat` + `.messages.parse()`); wired into every in-scope ambient channel/group message via `apps/server`. As of chunk 3.4a-i, a High-band score additionally drives ticket-draft composition (see **High-band auto-draft**, below); as of chunk 3.4c, a Low-band score additionally logs a real review-queue row (see **Review queue**, below); as of chunk 3.4b-i, a Mid-band score additionally posts a real confirming question (see **Confirming question (Mid-band)**, below). Gated by the same per-persona monthly cost cap as the DM chat-reply path (`checkCostCapAndAlert`, `haikuCostUsdMicros`, BUILD_PLAN 2.6a/2.6b) — a real, billed Anthropic call regardless of which model or call site it's on.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |

| **High-band auto-draft**             | The action a High-band confidence score triggers (`docs/VISION.md` §5.2) — an LLM composes a link-only ticket draft (title + body, no invented detail beyond the source message) from the classified message. Built at `BUILD_PLAN.md` chunk 3.4a-i as `composeTicketDraft` (`packages/agents`, Claude Sonnet 5, `zodOutputFormat` + `.messages.parse()`), wired into `handleAmbientChannelMessage` (`apps/server`). Real, live end to end as of chunk 3.4a-iii: gated by the **Situational-appropriateness gate** (below) in front of composition, the draft is posted as a real in-thread Slack message (`postMessage`), a `pending_ticket_drafts` row is persisted keyed on the real posted message's own `ts`, and the 📦/🔁/✅ reaction legend (park to Backlog / redo / commit as a ticket) is seeded with real `reactions.add` calls (`addReaction`, `packages/slack`) — no longer shadow/log-only. Respects the 2.7a operating-rhythm guard (`evaluateOperatingRhythm`) — 2.7a's own settled off-hours policy names "intake drafts" specifically as deferring to the next core-hours window, unlike the classification step itself, which that guard doesn't gate. Gated by its own fresh per-persona cost-cap check and `sonnetCostUsdMicros` accounting, same mechanism as every other real, billed call site in this cascade. Its cost-cap/operating-rhythm guard and the **Situational-appropriateness gate** below moved to a shared `standing-proactive-guards.ts` at chunk 3.4b-i, once the Mid-band confirming question needed the exact same two checks.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Confirming question (Mid-band)**   | The action a Mid-band confidence score triggers (`docs/VISION.md` §5.2) — a short, fixed-template confirming question, not an LLM-composed one (Alex confirmed via `AskUserQuestion`: no new billed call site, and the wording is cheap to change later since it's a plain string, not a schema/architecture choice). Ask-side built at `BUILD_PLAN.md` chunk 3.4b-i as `composeAndPostConfirmingQuestion` (`apps/server`): gated by the same cost-cap+operating-rhythm guard and **Situational-appropriateness gate** the **High-band auto-draft** uses (both extracted to shared `standing-proactive-guards.ts` this chunk), then posts the question in-thread on the source message, persists a `pending_confirming_questions` row (`packages/core`, workflow-object shape with a `resolvedAt`/CAS claim, same as `pending_ticket_drafts` — not `review_queue`'s append-only shape, since a question genuinely resolves once), and seeds a 👍 (yes, draft it) / 👎 (no) reaction legend — deliberately distinct from the High-band 📦/🔁/✅ legend so a later dispatch can tell them apart by reaction name alone. Short-names verified against Slack's own published event reference (its own example payload shows `"reaction": "thumbsup"`), not guessed from `iamcal/emoji-data`'s primary/alias split, which would have given the wrong answer (`+1`/`-1`) here. The answer-side is real as of chunk 3.4b-ii: `handleReactionAdded`'s dispatch (`apps/server`) classifies a reaction against both legends in order (`classifyReactionOutcome` first, then the new `classifyConfirmingQuestionOutcome`, `packages/slack` — a pure, `Map`-based `thumbsup`/`thumbsdown` → `yes`/`no` mapping, same external-string-lookup reasoning as its sibling) and dispatches to a sibling `dispatchConfirmingQuestionOutcome` function mirroring the pre-existing draft-outcome dispatch's own lookup → null-check → resolved-check → outcome-switch shape. 👍 (`draftFromConfirmingQuestion`) checks the cost cap _before_ claiming the question — a halted persona leaves it unresolved for a later retry — then reuses the **High-band auto-draft**'s own `postAndPersistDraft` _directly_ (not reimplemented; that function's params were narrowed to a minimal structural type so a reaction-outcome context, which has no `channelType`/`userId` to build a full inbound message with, could call it). 👎 (`logConfirmingQuestionAsNo`) logs to **Review queue** with `outcomeReason: 'mid-no'` — a real widened value now, not a placeholder (chunk 3.4b-ii's own `0009_widen_review_queue_outcome_reason.sql` migration). Live-verified end to end against real Anthropic/Slack/Postgres: a real 👍 reaction produced a real ticket-draft Slack message whose own `thread_ts` was confirmed (via a direct API call) to match the _original_ seed message, not the confirming question's own posted message. Silence/no-response detection (a scheduled sweep, since the codebase has no cron infrastructure yet) is deferred further still, to chunk 3.5 — `outcomeReason: 'mid-silence'` already exists in the widened constraint, pre-seeded ahead of that future writer. |
| **Review queue**                     | The action a Low-band confidence score triggers (`docs/VISION.md` §5.2's "nothing is silently eaten" backstop) — a plain append-only log row, not a workflow object (no resolved/claimed state, unlike **Reaction-gate outcomes**' `pending_ticket_drafts`, below). Built at `BUILD_PLAN.md` chunk 3.4c as a new `review_queue` table + `createReviewQueueEntry` (`packages/core`), wired into `handleAmbientChannelMessage`'s Low-band branch (`apps/server`, `logToReviewQueue`) — captures the source message text plus the Stage 1 classifier's own confidence/reasoning, so a human has real context, not just a bare score. `outcomeReason` records why a row exists: `'low-confidence'` is chunk 3.4c's own write; `'mid-no'` is chunk 3.4b-ii's own write (`logConfirmingQuestionAsNo`, see **Confirming question (Mid-band)**, above), once its Mid-band confirming question resolves to "no"; `'mid-silence'` is a placeholder value pre-seeded into the `CHECK` constraint ahead of a real writer — chunk 3.5's own future sweep is what logs it, once an unanswered confirming question passes some age threshold. Chunk 3.4c originally pre-seeded a single placeholder value, `'mid-no-response'`; chunk 3.4b-ii's own `0009_widen_review_queue_outcome_reason.sql` migration _replaced_ it with the two distinct values above (not an additive change), per 3.4c's own DA-review-flagged question — "no" and "silence"/timeout need to stay separately identifiable for 3.5's own human-eyeballing sweep. No consumer reads this table yet — chunk 3.5's own future sweep ("a formatted Slack post... for human eyeballing") is what lists these rows.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Situational-appropriateness gate** | The check VISION §9 requires before any standing-proactive action, named after the Viktor incident (a competitor bot reacting to a layoff announcement with a skull emoji) — a system that can act without being asked can act at the wrong moment. Built minimally at `BUILD_PLAN.md` chunk 3.4a-iii as `evaluateSituationalAppropriateness` (`packages/agents`, Claude Haiku 4.5, `zodOutputFormat` + `.messages.parse()`, returning `{appropriate: boolean, reasoning: string}`): an LLM content classifier over the source message text, checking for a serious/sensitive human situation (layoffs, death, crisis, grief, conflict) versus ordinary work chat — Alex confirmed via `AskUserQuestion`, not a keyword list or external toggle. **Fails CLOSED** on any gate failure (an API error, not just `appropriate: false`) — the opposite of `checkCostCapAndAlert`'s fail-open design, since an uncertain safety signal should block the risky action, same "better to wrongly rest once" reasoning as 2.7a's bank-holiday guard. Gates the **High-band auto-draft** (above) only, not the reaction-outcome dispatch below — Alex confirmed via `AskUserQuestion` that a human's own reaction is a response to the bot, not the bot acting unprompted, same distinction 2.7a's core-hours guard already draws for DM replies. 8.5 extends this to every standing-proactive surface; this chunk only wires the first one.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Reaction-gate outcomes**           | The three deterministic actions the 📦/🔁/✅ reaction legend maps to (`docs/VISION.md` §5.4's trust-erosion rule: routing decisions with real consequences stay off the LLM layer, not just the drafting step). Built at `BUILD_PLAN.md` chunk 3.4a-ii: `classifyReactionOutcome` (`packages/slack`, a pure Slack-reaction-short-name → outcome mapping — short-names verified against the real `iamcal/emoji-data` database, not guessed: ✅ is `white_check_mark`, 🔁 is `repeat`, 📦 is `package`), a `pending_ticket_drafts` table + repository (`packages/core`, the "parent-message state" tracking which draft a Slack message's reactions belong to — `resolvePendingTicketDraft`'s atomic claim, same CAS shape as chunk 1.3's `claimTicket`, guards against a reaction double-firing the same outcome twice), and three outcome-path functions (`apps/server`'s `reaction-outcome-actions.ts`): ✅ commits the draft as a real `Brief` ticket, 📦 parks it to an untriaged `Backlog` ticket (both `projectKey: 'chief-clancy'`, VISION §3.4's single-project scope; `severity: 'Medium'`, a fixed placeholder Alex confirmed via `AskUserQuestion` until a real triage signal exists), and 🔁 recomposes a fresh draft from the _original_ source message via `composeTicketDraft` (its own fresh cost-cap check and `sonnetCostUsdMicros` accounting) and updates the pending draft in place, leaving it open for a further reaction. ## Review & process (`docs/DEVELOPMENT.md`, `docs/DA-REVIEW.md`)

| Term                         | Definition                                                                                                                                                                                                                                                                                          |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Chunk**                    | A single unit of `BUILD_PLAN.md` work — roughly one PR, one sitting, one reviewable idea, per `BUILD_PLAN.md`'s own sizing-discipline note.                                                                                                                                                         |
| **Stage**                    | A group of chunks in `BUILD_PLAN.md` ending in something observable working end-to-end.                                                                                                                                                                                                             |
| **[GATE] chunk**             | A chunk that needs an Alex conversation before it can proceed — flagged explicitly in `BUILD_PLAN.md`, never guessed past.                                                                                                                                                                          |
| **DA review**                | Devil's-advocate review, dispatched from a fresh context, checked against `docs/DA-REVIEW.md` + `docs/CONVENTIONS.md` + `docs/RATIONALIZATIONS.md` + `docs/REVIEW-PATTERNS.md`.                                                                                                                     |
| **spec-grill**               | Two-phase adversarial review of a spec/plan/rationale doc before code moves, per `docs/DEVELOPMENT.md` §Two-phase grill discipline.                                                                                                                                                                 |
| **copilot-surrogate**        | Factual-claim reviewer, mandatory on `fix(docs)` commits and on any PR touching a blast-radius doc, exceeding 50 LOC, or making a factual claim about the rest of the repo or an external library (`docs/decisions/REVIEW-GATE-DISCRETION.md`), reading the full PR diff at HEAD (not diff-scoped). |
| **Do-not-touch list**        | Surfaces requiring Alex's explicit approval before editing (persona prompts, `docs/CEREMONIES.md`, `docs/VISION.md` §2/§14/§4.1) — `CLAUDE.md` §Non-obvious constraints.                                                                                                                            |
| **NOTICED BUT NOT TOUCHING** | The self-review discipline of listing an out-of-scope improvement rather than fixing it inline (`docs/SELF-REVIEW.md`).                                                                                                                                                                             |
| **Blast-radius doc**         | A policy-tier doc requiring Alex's merge (not self-merge), listed in `docs/GIT.md`. This doc, `ARCHITECTURE.md`, and `decisions/` are deliberately NOT on that list, matching chief-clancy's own `docs/INDEX.md` precedent (its own reference-tier docs are auto-merge-eligible).                   |
| **ADR**                      | Architecture Decision Record — the general industry term for a short, dated record of an architectural choice and its rationale.
